# New One API

一个基于 Cloudflare Workers 构建的高性能、Serverless 的大模型 API 转发与代理分发系统。它不仅支持多供应商接入与完善的 Token 管控，还提供企业级的高可用架构设计（如多路负载均衡、自动重试和模型降级无缝切换）。本项目完全部署于边缘网络，并且自带功能完善的可视化管理后台。

## 🌟 通用核心功能介绍 (General Features)

本项目通过将多个不同供应商提供的大语言模型（LLM）API 进行封装和转发，对外提供统一、标准的 OpenAI 接口格式调用（如 `/v1/chat/completions`、`/v1/messages` 等）。

### 1. 多渠道模型聚合 (Multi-Channel Aggregation)
- 兼容丰富的上游供应商服务，原生支持 **OpenAI**、**Azure OpenAI**、**Claude** 等。
- 支持协议级别的双向转换（例如智能将 Claude 等模型转换为标准的 OpenAI 请求和响应格式返回给客户端），实现各种复杂渠道的“无缝对接”。

### 2. 精细的凭据与额度划分 (Token & Quota Management)
- 使用 `api_token` 发放独立的调用凭据。
- **额度管控**：为每个 Token 设定独立的额度上限（`total_quota`），避免滥用导致的费用激增，并能实时统计消费明细（精确到消耗的 Input/Output Token 数量）。
- **权限隔离**：可指定不同的 Token 只允许访问特定范围的频道 (Channels)，达到多租户与安全管控的目的。

### 3. 可视化的 Web 管理后台 (Admin UI)
内置了由 React 构建的美观交互面板，包括以下核心模块的图形化管理：
- **仪表盘 (Dashboard)**：宏观查看调用情况。
- **渠道管理 (Channels)**：添加、启用和修改不同的上游 API 供应商节点。
- **令牌管理 (Tokens)**：生成、冻结或刷新用户 API 秘钥。
- **动态寻址 (Deployment Mapper) 与计费 (Pricing)**：可以将用户发出的标准请求模型（如 `gpt-4`）重新映射成后台真实调用的部署名称，并定义具体的倍率基准。

## 🛡️ 高可用调度：智能重试与模型降级链 (Failover Mechanism)

为了应对大模型在使用中常出现的接口限流（Rate Limit）或服务端超时宕机，网络稳定性变得尤为重要。本项目特设了一套详尽的容灾调度机制（Failover）。

### 核心运作机制：

#### 1. 自动甄别异常与并发请求分级
当请求发送至上游引发阻塞时，代理层将精确判断 HTTP 状态码。只有当触发非应用层级的阻断——即 **服务器异常 (5xx)** 或 **并发限流 (`429 Too Many Requests`)** 时，系统才会主动拦截错误并触发故障转移；而对于诸如参数写错引发的 `400 Bad Request` 等逻辑错误，服务将快速失败 (Fail-fast)，向客户端返回报错，避免无意义的等待与重试。

#### 2. 同级节点的负载均衡与智能重试设定
- **优先级 (Priority)**：各渠道配置可赋予独立优先级索引（数值越大优先级越高）。服务将优选快速或价格更低的节点池进行请求。
- **重试上限限定 (模型最大尝试次数 / `max_retries_per_model`)**：在相同的模型名称下，为防止无休止地对失效服务轮询，可在管理端配置**每个模型级别的最大尝试次数**（默认通常配置为 3 次）。服务将从优先队列中提取节点，若请求出错且不满足成功响应，将会在配额允许次数内平滑流转到同个模型的下一个备用通道。
- **请求打散**：如果多个节点配置了相同的优先级，系统底层的路由分发将会对其进行随机乱序（Random Shuffle），借此实现流量侧的负载均衡。

#### 3. 优雅的保底方案：模型降级链 (Model Fallback Chains)
面对某类高级模型全网拥堵或集体无响应（例如 `gpt-5.4` 完全不可用且达到了上文提到的大尝试次数指标），系统将激活最后的安全网：**模型降级链**。
- **自定义降级流 (Chains)**：管理员事先配置如 `["gpt-5.4", "gpt-5.2", "gpt-5.1"]` 的降级链路数组。
- **透明式降级替补**：当首选模型（`gpt-5.4`）内所有相关渠道穷尽均告失败后，原代理逻辑不向外直接抛出 Http 502 错误，而是将用户请求里携带的 Model 指令“降格”修改，向链路内的下一位（`gpt-5.2`）再次发起挑战。直至有模型响应或整个链条均遭摧毁为止。
- 这一过程对发起调用的第一方应用是**完全透明**的，有效保障了高可用 AI 服务所需要的 “Always On” (永远在线) 的硬性业务指标！

## 💻 本地开发与体验

### 后端运行环境 (Worker)
执行以下命令以唤起 Wrangler，运行并监听代理分发的云服务：
```bash
# 复制示例环境变量
cp wrangler.toml.template wrangler.toml

# 启动本地代理与调度环境
pnpm dev
```

### 前端管理面板构建 (Admin UI)
前往 `frontend` 目录构筑供 Worker 输出的静态面板资产：
```bash
pnpm -C frontend build
```

## 📝 贡献代码与反馈
在增加新的提供商通道支持（Provider）时，请按照 TypeScript 中 `src/providers` 预设代理契约格式添加。并在界面修改时保证同步构建到 `/public` 目目录下。
